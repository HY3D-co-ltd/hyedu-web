# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Static rebuild of **hyedu.kr** (한양미래연구소, an education platform for K–12 AI / robot coding / maker / STEAM programs). The goal is to replace an external BBS-based site with a self-hosted Next.js site optimized for SEO/AEO, deployable to GitHub Pages.

The detailed design spec lives at `docs/superpowers/specs/2026-04-14-hyedu-website-rebuild-design.md` — consult it for scope, page inventory, content mapping, and SEO strategy before making non-trivial changes.

## Commands

- `npm run dev` — Next.js dev server (port 3000).
- `npm run build` — production build; emits a fully static site to `out/` because of `output: 'export'` in `next.config.ts`.
- `npm run start` — serves via `next start` (only useful for non-exported testing; prefer `serve` for parity with production).
- `npm run serve` — serves the exported `out/` directory on port 3000 via `http-server`. Use this to validate what GitHub Pages will actually ship.
- `npm run lint` — ESLint (`eslint-config-next`).

Type-checking is done via `tsc --noEmit` implicitly through the build; there is no separate script. Run `npx tsc --noEmit` to check types without building.

### Data crawling / migration scripts

- `npx tsx scripts/crawl-board.ts` — pulls BBS posts from `www.hyedu.kr` and writes them to Firestore under `boards/{reviews|events|gallery}/posts/bbs_{id}`. Requires `GOOGLE_APPLICATION_CREDENTIALS` env var pointing to a Firebase Admin service-account key. Regexes in `extractPostSummaries` / `extractPostDetail` target the legacy BBS HTML shape and may need tweaking if that site changes.
- `npx tsx scripts/download-images.ts` — downloads assets from the legacy site into `public/images/`.

No test suite is configured.

## Architecture

### Static export + i18n routing

The site is a **fully static export** (`next.config.ts`: `output: 'export'`, `trailingSlash: true`, `images.unoptimized: true`). This constrains what you can use: no server actions, no runtime API routes, no `next/image` optimization pipeline, and every dynamic route must declare `generateStaticParams`.

Routing is locale-prefixed via `next-intl`:

- `src/middleware.ts` runs `next-intl` middleware matching `/` and `/(ko|en)/:path*`.
- `src/i18n/routing.ts` declares locales `['ko', 'en']` with `ko` as default.
- All pages live under `src/app/[locale]/…`. The `[locale]/layout.tsx` calls `setRequestLocale(locale)` and wraps children in `NextIntlClientProvider` with messages from `src/messages/{ko,en}.json`.
- The top-level `src/app/layout.tsx` is intentionally minimal — locale-aware HTML/head/body lives in `[locale]/layout.tsx`.

When adding a new page, put it under `[locale]/`, and if it has dynamic segments, both the slug **and** the locale must be enumerated in `generateStaticParams` (see `programs/[slug]/page.tsx`).

### Content model: static data vs. Firestore

Content falls into two buckets, and knowing which is which is essential:

1. **Static TypeScript data** in `src/data/*.ts` (programs, camps, youth-clubs, online-courses, special-lectures, booths, partners, history, cancellation-policy, navigation). These are imported at build time and drive statically-rendered pages. `src/sitemap.ts` iterates over `programs` to emit per-slug sitemap entries. Types live in `src/types/index.ts` (`Program`, `YouthClub`, `OnlineCourse`, …).
2. **Firestore-backed content** for things that non-devs need to edit without a rebuild: BBS-style boards (`reviews`, `events`, `gallery`) and homepage popups. Read helpers are in `src/lib/board.ts` and `src/lib/popup.ts`, using the client SDK initialized in `src/lib/firebase.ts` (config comes from `NEXT_PUBLIC_FIREBASE_*` env vars). Pages that consume Firestore are **client components** (`'use client'`) that fetch in `useEffect` — they have to be, because the static export has no server runtime.

Do not try to read Firestore in a server component; it will run at build time and bake a snapshot into the static HTML, defeating the point.

### SEO / AEO

`src/components/seo/JsonLd.tsx` exports `OrganizationJsonLd`, `CourseJsonLd`, `FAQJsonLd`, and `BreadcrumbJsonLd`. `OrganizationJsonLd` is mounted once in `[locale]/layout.tsx`; the others are per-page. Program detail pages emit `CourseJsonLd` + `FAQJsonLd` + `BreadcrumbJsonLd`. The sitemap is generated by `src/app/sitemap.ts` at build time and includes hreflang alternates for `ko`/`en`.

When adding a new page type with structured data potential, add a new JsonLd helper rather than inlining `<script type="application/ld+json">` in the page.

### Layout composition

`[locale]/layout.tsx` always renders `Header`, `Footer`, `Popup`, and `OrganizationJsonLd`. Page components render only their own content (hero + sections). Shared section components live in `src/components/sections/` and are designed for the homepage — don't import them into inner pages unless intentionally reusing a section.

### Styling

Tailwind with custom tokens in `tailwind.config.ts`: `primary` (blue #2563eb), `secondary` (amber), `accent` (green), `point` (teal #336666), and a `scroll` keyframe/animation used for marquee-style partner logos. Font family is `Pretendard` (loaded externally — check `globals.css`).

## Conventions

- Path alias `@/*` → `src/*`. Use it in imports.
- Copy is bilingual. Every new content item needs both `foo` (ko) and `fooEn` (en) fields on the type, and every UI string should either come from `messages/{ko,en}.json` via `useTranslations`/`getTranslations` or be gated with `locale === 'ko' ? … : …`.
- When adding a page under `[locale]/…`, also add it to the `staticPages` array in `src/app/sitemap.ts` so it ships in the sitemap with hreflang alternates.
- When adding a program, update `src/data/programs.ts` — the sitemap and program detail routes pick it up automatically via `generateStaticParams`.

## What lives where (quick map)

- Page routes → `src/app/[locale]/…`
- Static content → `src/data/*.ts` + types in `src/types/index.ts`
- Firestore access → `src/lib/{firebase,board,popup}.ts`
- Shared UI → `src/components/{layout,sections,ui,seo}/`
- i18n strings → `src/messages/{ko,en}.json`
- Design/spec of the rebuild → `docs/superpowers/`
